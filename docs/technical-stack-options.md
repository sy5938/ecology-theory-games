# 光之林：技术栈与重构架构草案

状态：**已用于 V2 可玩原型，正式产品选型仍可在试玩后调整**
日期：2026-07-17

## 这次试玩推翻了什么

当前 `prototype/index.html` 只应视为一次失败但有价值的技术验证，不继续在其上堆功能：

- 点击按钮后整局状态瞬间跳变，玩家看不到连续生态过程；
- 地图个体没有 hover/click 状态检查，空间信息无法参与决策；
- 固定 12 回合把树木生活史压缩成了短周期按钮循环；
- 全图 pulse 动画没有表达具体生态事件，应该移除；
- 光照虽在绘制时重算，但遮阴公式把归一化树高再次除以最大高度（米），使林冠死亡后的变化几乎不可见；
- 缺少选中个体、玩家种群和群落尺度的时间序列；
- 图例同时承担物种、策略、阶段等信息，认知负担过高；
- 没有“密闭林冠”等可比较的初始情景。

## 技术栈候选

| 方案 | 建议组合 | 优点 | 主要代价 | 适用判断 |
| --- | --- | --- | --- | --- |
| Pygame | Python + Pygame + 自绘 UI/图表 | Python 生态模型衔接直接；实时循环和鼠标事件容易实现；适合本地研究沙盒 | 浏览器分享困难；状态面板、响应式布局和动态图表需要大量自绘或额外依赖；打包分发成本较高 | **能做，但不推荐作为面向所有专业背景的首发版本** |
| Phaser Web | Vite + TypeScript + Phaser 3 + Apache ECharts | 自带场景、更新循环、Game Object 和统一指针输入；浏览器即开即玩；DOM 面板和动态图表成熟 | 需要把现有单文件原型重写为模块；生态模型需用 TypeScript 实现 | **推荐** |
| PixiJS Web | Vite + TypeScript + PixiJS + Apache ECharts | 渲染层轻、可控制大量点位，适合科学可视化 | 游戏循环、场景、状态和交互组织需要自己搭建 | 若后续个体数量远高于 MVP，可替代 Phaser |
| Godot | Godot 4 + GDScript | 游戏编辑器、动画、输入和桌面打包完整，长期可做成更像游戏的产品 | 三天内学习与调试成本较高；科学图表和网页周边 UI 不如 Web 直接 | 适合后续独立游戏化，不适合本轮快速重构 |

## 当前推荐

采用：**Vite + TypeScript + Phaser 3 + Apache ECharts**，不使用 React，不设后端。

该组合已经落地到 `prototype-v2/`：模拟核心与 Phaser 渲染分离，ECharts 读取模拟历史。三个早期布局已根据试玩反馈收敛为一个统一仪表盘，不再保留运行时布局切换。

- Phaser：地图、个体点位、hover/click、摄像机、连续更新循环；
- TypeScript：生态模拟、规则和数据结构；
- ECharts：时间序列和当前分布图；
- HTML/CSS：状态面板、初始条件、暂停/倍速和策略操作；
- CSV/JSON：物种与情景参数；现有 R/Python 只做离线参数估计，不进入游戏运行时；
- Vitest：固定随机种子的模拟规则测试；
- 静态部署：无需登录和安装，便于 summer school 现场分享。

选择 Phaser 的直接原因：官方输入模型允许给 Game Object 设置交互区域并响应 pointer 事件；Scene 同时管理更新循环、显示列表和输入。ECharts 可独立接收持续更新的数据，不需要把图表塞进游戏画布。

## 目标架构

```text
物种/情景参数
      ↓
Simulation Core（纯 TypeScript、固定生态时间步、可复现随机数）
      ↓ 产生 birth / growth / death / canopy-change 等事件
 ┌───────────────┬────────────────┐
 ↓               ↓                ↓
Phaser 地图      状态检查面板      ECharts 指标图
个体与光照场      选中个体/格点      时间序列/当前分布
```

建议目录：

```text
src/
  simulation/    # 个体、生活史、竞争、扰动、胜负判定
  game/          # Phaser 场景、点位、光照纹理、输入
  ui/            # 状态面板、控制器、ECharts
  data/          # 物种和初始情景 JSON
  tests/         # 固定种子的模拟回归测试
```

关键边界：

- **渲染帧**与**生态时间步**分离：画面可保持流畅，生态状态按固定频率更新；
- 默认生态时间步建议 500 ms，支持暂停、1×、2×，具体数值在玩法确定后校准；
- 动画只解释事件：种子飞散、阶段变化、死亡淡出、林窗变亮；移除全图脉冲和等待动画；
- 光照场只在林冠结构改变时重算并缓存，不在每个绘制帧遍历全部树木；
- 所有规则先写入 Simulation Core，地图只读取状态，不自行决定生死；
- 使用固定随机种子，保证同一初始条件和策略可以复盘比较。

## 必须支持的交互

- hover 个体：高亮并显示简短摘要；
- click 个体：固定右侧状态卡，显示物种、策略、阶段、年龄、树高、胸径、局部光照和近期变化；
- click 空地：显示该格的光照、林冠覆盖和邻域密度；
- 图例只编码两层信息：颜色表示喜阳/喜阴/广适，外圈表示“自己”；生活史阶段主要由点大小和状态卡说明；
- 初始情景至少预留“密闭林冠森林”接口；其他情景后续再定；
- 游戏连续运行，不再以 12 次点击作为寿命；
- 连续运行满三分钟时暂停并展示群落结局，玩家可以继续或重新开始；玩家物种灭绝时立即结束；
- 所有物种和个体都可检查，但生命史投资控件只绑定玩家所选物种；玩家自己的幼苗/幼树允许一次拖拽移栽；
- 游戏界面采用稳定的地图/分析双栏：地图为主视图，投资组合横跨下方；不使用覆盖地图的浮层或全局方向键切换；
- 操作区必须支持生长、繁殖、储备总和为 100% 的持续配置，并显示当前组合何时生效；基础维持成本不进入配置盘；
- 模拟核心分别记录碳收入、基础维持成本、当期可投资资源和跨时间累积的碳储备；
- 繁殖投资只控制投入量，不提供额外的播种、远距扩散或种子库模式按钮；
- 随机事件可以显示模糊预警，但不增加专用技能或事件操作按钮；

## 图表边界

“动态变化”和“直方图”需要区分：

- **时间序列折线图**：回答某个个体或种群如何随时间变化；
- **直方图**：回答当前时刻树高、年龄或局部光照如何分布。

第一版建议只做一个可切换图表区：

1. 选中个体：树高、胸径、局部光照的时间序列；
2. 玩家种群：总个体、幼苗、幼树、成树数量的时间序列；
3. 当前群落：玩家与其他物种的树高或光照分布直方图。

## 暂不决定

以下内容必须等核心玩法确定后再编码：

- “占据整个群落”按个体数、空间、林冠面积还是生物量判定；
- 共存与单种优势都已确定为可能结局，但结局分类、得分和触发阈值仍待确认；
- 密闭林初始条件中，玩家从种子、幼苗还是混合阶段开始。
